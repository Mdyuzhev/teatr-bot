## О проекте

Telegram-бот, который собирает афишу московских театров из открытых источников
(KudaGo API, RSS, Culture.ru), хранит в PostgreSQL и по запросу пользователя
выдаёт готовый дайджест.

**Ключевой принцип**: данные и дайджесты генерируются по расписанию, независимо
от пользовательских запросов. Пользователь всегда читает из кэша — вызов LLM
происходит только планировщиком, не на запрос пользователя.

---

## Архитектура генерации дайджестов (ВАЖНО)

### Почему не «запрос → LLM»

Наивная схема (каждый пользовательский запрос вызывает Haiku) расточительна:
10 пользователей за день = 10 одинаковых вызовов LLM на одни и те же данные.
Афиша не меняется каждый час — она обновляется раз в сутки.

### Правильная схема: планировщик + кэш

```
06:00  Сбор данных KudaGo (1 раз в сутки)
06:30  Сбор RSS новостей театров
07:00  Генерация стандартных дайджестов через Haiku (4 вызова LLM в день):
         → 'today'    (сегодня)
         → 'tomorrow' (завтра)
         → 'weekend'  (ближайшие Сб-Вс)
         → 'week'     (текущая неделя Пн-Вс)
       Результаты сохраняются в таблицу digests с expires_at = NOW() + 24h

Пользователь /today   → SELECT из digests WHERE period_key='today' AND expires_at > NOW()
                       → мгновенный ответ, 0 токенов
Пользователь /weekend → SELECT из digests WHERE period_key='weekend' AND expires_at > NOW()
                       → мгновенный ответ, 0 токенов

На следующий день 07:00 → INSERT ... ON CONFLICT DO UPDATE → кэш обновлён
```

### Кастомный период (пользователь вводит даты вручную)

```
/digest 20.03 - 25.03
  → period_key = '2026-03-20:2026-03-25'
  → есть в digests И expires_at > NOW()?
      ДА  → отдаём из кэша (0 токенов)
      НЕТ → вызываем Haiku, сохраняем с expires_at = NOW() + 24h
```

Таким образом, даже нестандартные периоды кэшируются на 24 часа.

### Итог по расходу токенов

- Стандартные периоды: **4 вызова Haiku в сутки** (фиксировано)
- Нестандартные периоды: **1 вызов на уникальный период в 24 часа**
- Пользовательские запросы: **0 вызовов LLM**

## Структура БД

```sql
theaters     — театры (slug UNIQUE — ключ дедупликации)
shows        — спектакли (slug UNIQUE — ключ дедупликации)
show_dates   — даты показов (UNIQUE: show_id + date + time + stage)
rss_news     — новости RSS (url UNIQUE)
digests      — кэш дайджестов (UNIQUE: period_key + date_from + date_to)
bot_events   — журнал событий бота
```

### Таблица digests (ключевая)

```sql
digests (
    period_key   VARCHAR(100)   -- 'today','tomorrow','weekend','week' или '2026-03-20:2026-03-25'
    date_from    DATE
    date_to      DATE
    content      TEXT           -- готовый HTML-текст для Telegram
    shows_count  INTEGER
    model        VARCHAR(100)   -- 'claude-haiku-4-5-20251001'
    generated_at TIMESTAMPTZ
    expires_at   TIMESTAMPTZ    -- generated_at + 24h, индексировано
    UNIQUE(period_key, date_from, date_to)
)
```

## Telegram-команды

| Команда | Источник данных |
|---------|-----------------|
| `/today` | digests WHERE period_key='today' |
| `/tomorrow` | digests WHERE period_key='tomorrow' |
| `/weekend` | digests WHERE period_key='weekend' |
| `/week` | digests WHERE period_key='week' |
| `/digest` | Inline-кнопки → одна из выше или кастомный период |
| `/theater [название]` | shows + show_dates (прямой SQL, без LLM) |

## Важные архитектурные решения (НЕ менять без обсуждения)

**1. LLM вызывается ТОЛЬКО планировщиком, не по запросу пользователя.**
Telegram-команды читают готовый текст из таблицы `digests`. Это принципиально.

**2. Кастомный период = кэш на 24 часа.**
Если пользователь запрашивает нестандартный период — сначала проверяем `digests`,
только если нет или устарел — вызываем Haiku и сохраняем результат.

**3. TTL кэша = 24 часа.**
`expires_at = generated_at + INTERVAL '24 hours'`. Планировщик обновляет
стандартные дайджесты каждое утро в 07:00, тем самым автоматически обновляя кэш.

**4. KudaGo — 1 раз в сутки.**
Данные афиши не меняются поминутно. Ночной сбор + утренняя генерация = свежие
данные к началу дня.

**5. asyncpg, raw SQL, без ORM.**

**6. Порт 5435 для PostgreSQL** — не конфликтует с moex-postgres на 5434.

**7. Claude API только из `brain/digest_builder.py`.**
Ни коллекторы, ни модули БД, ни команды Telegram не вызывают Anthropic API.

## Критические ограничения

- **НЕ** вызывать Haiku из Telegram-команд — только из планировщика и кэш-миссовых путей
- **НЕ** вызывать Claude API из collectors/ или db/ — только из brain/
- **НЕ** строить Docker образы локально на сервере (риск OOM)
- **НЕ** использовать ORM — только raw SQL через asyncpg
- **НЕ** запрашивать KudaGo чаще 1 раза в 6 часов
- **НЕ** хранить персональные данные пользователей Telegram
